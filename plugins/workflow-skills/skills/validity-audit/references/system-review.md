# 第 3 段(條件觸發):系統體檢 — 當受審對象是「已部署系統 / 每日管線 / 專案結案」

**為什麼存在**:2026-07-06 使用者抓到的盲區——VA 三次審過 worldcup-2026 部署系統,
每次都在否證「宣稱」,從沒問「沒人宣稱、但錯了的東西」。結果:唯一不可重生的
predictions_ledger 無備份、論文 .tex 從未編譯過、07-20 自動判定檢定力趨近零——全靠
使用者要求一次「人工專業評審」才浮現。**否證宣稱 ≠ 系統健康。**

**觸發條件**:受審對象含以下任一 → 前兩段照跑,**加跑本段**——
已上線 cron/daemon、對外推播、寫入累積性 ledger/資料、專案結案(要放著自轉)、交付物(論文/報告/網站)。
純研究宣稱(一個回測結論)不觸發本段。

**執行方式**:下列 7 項逐項答,每項附「查了什麼指令/檔案」+ PASS/FLAG。
可併入獨立 reviewer 的任務,但 prompt 要明說:「你的威脅模型不是『宣稱是假的』,
而是『**沒人宣稱過、但會咬人的東西**』」(completeness-critic 視角)。

## S1 不可重生資產與備份
列出系統中「**重算不出來**」的檔案:point-in-time 預測/決策 ledger、外部 API 快照(配額限制)、
人工標註、憑證以外的一次性資料。每個問:①有第二份副本嗎(另一磁碟/遠端)?②在 git 裡嗎?
③repo 本身有 off-machine 備份嗎(local-only git = 單點)?
**紅旗**:不可重生 + 被 gitignore + 單機唯一副本(= 磁碟一死全沒)。

## S2 交付物端到端實測
每個交付物問:「它被**以最終使用方式**實際執行過至少一次嗎?」——
.tex 進過編譯器?cron 指令從 cron 環境(非互動 shell、無 cwd)跑過?PDF 打開看過?
接收端(Telegram/email)真的收到過?安裝說明照著走過?
**紅旗**:「寫好了」被當成「能用了」;send/export 成功被當成內容正確(build123d 教訓同型)。

## S3 未來檢定的檢定力
對每個「之後會自動驗證/重跑」的承諾:預期樣本 n 多大?該 n 能偵測多大的效應?
不足就**現在寫下期望管理句**(「07-20 是儀式性檢查點,非判決」),別讓未來的自己誤讀。
**紅旗**:低檢定力的未來檢查被敘述成「到時就知道行不行」。

## S4 已知簡化清單(集中化)
把散落在註解/docstring 的簡化與近似**集中列表**:每項寫「偏誤方向 + 粗略量級 + 何時值得修」。
(例:和局晉級按勝率全比例分配→系統性略高估熱門,~2-3pp,PK 密集的賽事再修。)
**紅旗**:簡化只存在於寫它的人的記憶裡。

## S5 極限與配額
每個外部依賴/資源:API 配額(誰在扣?失敗會重試燒配額嗎?)、訊息長度上限(Telegram 4096)、
檔案無界成長(ledger/log 有 rotation 嗎?)、timeout。
**紅旗**:無監控的循環外部呼叫;「現在的量沒事」沒換算過「重用到更大場景」。

## S6 孤兒與雙入口
不在主路徑上的入口(舊 CLI、廢棄模組、實驗腳本):標明狀態(棄用橫幅/刪除/文件),
特別查「**兩個入口寫同一個檔案**」(worldcup 真踩過:predict.py 與 notify.py 曾用不同
schema 寫同一個 ledger)。
**紅旗**:跑了會污染正本的孤兒入口。

## S7 機會成本(一句話就好)
系統完成度 vs 下一單位工作的邊際價值:「再加功能」還是「該交付/發表/放著自轉」?
對照使用者的出口紀律(exit-log:瓶頸在出口不在產出)。本項不是工程判斷,是提醒
把答案**明說**——預設傾向是繼續蓋,說出來才會停。

## 回報格式
7 項各 ≤3 行(查法 + PASS/FLAG + 一句修法);FLAG 按 P1(會失去資料/誤導)/P2/P3 分級。
與前兩段的發現合併進同一份 verdict。
