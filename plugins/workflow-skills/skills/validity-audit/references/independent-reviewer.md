# 第 2 段:獨立 reviewer(★ 強制,不可省略)

用 **Agent 工具** spawn 一個 `general-purpose` subagent(沒參與建造)做對抗式複查。**兩輪,依序:**

1. **冷審(先做):** 只給碼 + headline 主張,**不給提示清單、不說作者發現了什麼**。要它自己推威脅模型、找問題。
   ——這才量得到「真獨立」(被指路的 reviewer 只會找你指的地方)。理想上**換不同模型家族(或人)**做冷審,因為同模型 reviewer 與建造者共用盲點。
2. **熱審:** 再給 Stage-1 發現 + bug 類別清單(「別重複、往更深挖」)。比對兩輪,差距 = 結果有多依賴 framing。

- 兩輪都要:**實際讀碼**(列關鍵檔)+ 可跑唯讀檢查,**別改檔/別跑長訓練**。
- 輸出:(a) 排序問題清單(file:line+機制+嚴重度)、(b) 可信/不可信 headline、(c) 單一最重要修正。

## Prompt 範本(填入專案路徑與宣稱)

> 你是獨立、對抗式的軟體工程審查者。審計 `<專案路徑>`,找出 correctness/coverage/metric inflation/bias 問題。
> 你沒建造它;找作者漏掉或合理化的瑕疵。
> 宣稱:<列 headline 主張 + 對應檔案>。作者已發現:<列已知問題>——別重複,往更深挖。
> 重點查:測試集污染(訓練/測試重疊)、覆蓋率灌水、benchmark 選擇偏誤、指標公式 bug(分母/平均方式)、
> 環境依賴(硬體/版本未固定)、CI 偽複製、多重比較未修正、部署現實落差。
> 讀實際的碼(列關鍵檔),可跑唯讀檢查但別改檔/別跑長訓練。
> 輸出:(a) 排序問題清單(file:line+機制+嚴重度)、(b) 可信/不可信的 headline、(c) 單一最重要修正。

> 註:冷審時**省略**「作者已發現」那行與「重點查」清單,只留前兩句 + 輸出格式。
