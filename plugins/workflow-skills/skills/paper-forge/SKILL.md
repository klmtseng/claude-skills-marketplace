# paper-forge — 論文/外部 repo → 內部能力的產線

**問題**:讀懂一篇論文不難,難的是把它「變成你系統裡真的在用的東西」。多數人(和多數 AI)停在「我跑起來了」(S5 驗證),就當結案——結果知識沒沉澱、沒接進任何地方,下次還要重讀一遍。本 skill 源自 11 個真實複刻研究的盤點:**應用→整合成功率只有 6/11**,失敗的全卡在「驗完就算完,沒有系統性的應用萃取步驟」。

**解法**:把「理解→重製→驗證→萃取→整合」釘成八階段產線(S1–S8),每階段有可機器判定的驗收條件。前段(S1–S5)路由到現成 skill,把力氣集中在別人都跳過、也是最高槓桿的後段(S6–S8):**誠實揭露、可移植原語、回搬接線**。

**適用**:arXiv 論文、GitHub repo、HuggingFace 模型、官方 agent repo、或純概念複刻——任何「外面有個東西,你想吃進來變成自己能力」的情境。
**不適用**:只想快速讀懂一篇論文說什麼(直接用深度研究工具就好,別套整條產線);純為引用而查一篇文獻(WebFetch 核對即可)。

## 開工前:先分類來源(決定 S1/S2 怎麼走)

| 來源類型 | S1 理解走 | S2 取得走 | 特別注意 |
|---|---|---|---|
| **arXiv / 論文** | 深度精讀 + 學術系譜;**引用/標題/作者先 WebFetch 核對**(見雷#1) | 通常無 code,S2=找有沒有官方/第三方參考實作 | 「論文說的」vs「別人解讀」要分清;參考實作自身可能不對齊論文(雷#7) |
| **GitHub repo** | `/code-recon`(架構/誰呼叫/blast radius) | `git clone`,原封保留 `original/` 或 `vendor/`(只讀) | 別一 clone 就改;先確認 License 能不能用/改/分享 |
| **HuggingFace 模型** | 讀 model card + 論文;確認授權與本機能不能載 | HF MCP 工具(先 authenticate)或 `huggingface-cli` | 本機算力限制提早確認(無 GPU/AVX2)——見雷#6 |
| **官方 agent / 產品 repo** | `/code-recon` + 讀 README/範例 | clone 到 `vendor/` clean-room,不與既有系統混 | 目標常是「抽範本」而非「跑起來」,S7 權重更高 |
| **純概念複刻** | 文獻 + 逆向(反編譯/wiki) | 自己搭最小骨架 | 沒有現成基準,S4 功能閘門要自己定義「對」長什麼樣 |

## 八階段產線

### S1 — 源精讀(理解到能複述機制,不是讀過)
- 讀到能一句話說出「核心機制是什麼、哪個部件真正產生了論文宣稱的現象」。
- **鐵則(雷#1)**:任何 arXiv id / 論文標題 / 作者名,寫進任何文件前先 WebFetch 核對;查不到就標「未查證」,**不准憑記憶填**(實案:曾有複刻研究引用了憑訓練記憶捏造的 arXiv 標題,被獨立審計抓到)。
- **導言-方法節對照**:導言/摘要講的機制故事可能 ≠ 方法節實際做的事(本 skill 首次 dogfood 一篇論文就抓到兩處:任務定義正文 vs 演算法框互斥、導言說非參數平均 vs 方法節是監督式管線)。宣稱清單與重製規格**從方法節萃取**;矛盾本身記錄為發現。
- **驗收**:寫得出 3–5 條「論文宣稱」清單(之後 S5 逐條驗的受測對象)。

### S2 — 取得與環境準備
- clone / 下載;**原始檔放只讀保留區**(`original/` 或 `vendor/`),絕不在上面直接改。
- **提早確認本機限制 + 繞法**(雷#6):無 GPU→CPU twin;無 AVX2→patch `target-cpu` / 換 BLAS;無 sudo→借用戶層裝法。半天死路多半來自沒先確認這個。
- **驗收**:`original/` 存在且未被改動;一份「本機限制 + 繞法」筆記。

### S3 — clean-room 重製
- **先 1:1 照做,不與既有系統混搭**(雷#3)。手寫最小可動版:NumPy prototype / CPU twin / headless PoC。
- 目的是拿到**乾淨基準**——沒有它,之後改壞了找不到哪裡壞。
- **驗收**:最小版能跑出論文的關鍵數字/圖(至少定性對齊 Fig.X)。

### S4 — 功能閘門 / 冒煙測試 → `/core-gate`
- 寫一跑即判對錯的驗收腳本(`acceptance.py` / `verify_*.py`)。
- **鐵則(雷#2)**:閘門要**量測可觀察行為**,不是「跑通就算對」(export 成功≠幾何正確;能執行≠數字對)。
- 若是確定性核心(引擎/回測/模擬),直接走 `/core-gate` 三閘門。
- **驗收**:一條指令、無互動、輸出可重現的 PASS/FAIL。

### S5 — 兩段式驗證 → `/validity-audit`
- **內審**(自查)→ **fresh-context 獨立 reviewer**(沒參與建造的 subagent 對抗式複查)。**兩段都做才算完**(雷#4:自己審自己會漏同型盲點)。
- **reviewer 要拿得到原始論文/repo(`original/`)**,不能只給你的摘要——否則它對「偏離論文」的判定可能建立在你的轉述上(首次 dogfood 實例:reviewer 沒讀 PDF,把忠於方法節的實作誤判為偏離,靠主對話對照原文才翻案)。reviewer 的裁決與原文有出入時,回原文仲裁,翻案記錄留在 `audit/`。
- 對 S1 的宣稱清單逐條判:證實 / 證偽 / 樣本不足。允許「查無結論」,不允許硬掰。
- 有隨機性的方向宣稱要 **≥3 seed**(雷#9);指標好看要做**端到端組合/下游測試**才算兌現(雷#8:forecast alpha≠portfolio alpha)。
- **驗收**:`/validity-audit` 報告,獨立 reviewer 已跑,宣稱清單逐條有裁決。

### S6 — 誠實揭露 / 撤回
- 驗證打掉哪條宣稱,就改文件/記憶,**把撤回過程保留在 `audit/`,不偷偷刪**。
- 負面結果是結果:寫清楚「試了什麼、為什麼沒成」——它常是下一個專案的起點。
- **驗收**:文件裡沒有被 reviewer 打掉卻還留著的宣稱;`audit/` 有撤回記錄。

### S7 — 洞見萃取(★ 差異化核心;產出「三件套」)
從「我重現了」進到「我抽出了可以帶走的東西」。產出固定三件套(範本見 `templates/insight.md`):

1. **機制圖**:核心機制拆成 `輸入 → 轉換 → 輸出`,標出「哪個部件是真正產生現象的關鍵」。
2. **可移植原語清單**:每條一行 = 名字 + 一句話定義 + 「脫離原論文脈絡還成立嗎?」。
   - 一篇論文抽 3–5 條乾淨原語即可,寧缺勿濫。
3. **整合觸發條件**:每條原語配一句「當我遇到 ___ 情境時,想起它」——這句之後直接進 memory 當召回鉤子。
- **驗收**:三件套齊全**且無空白佔位符**(範本裡的 `<尖括號>` 與空表格行全部填掉——`grep -c '<' INSIGHT.md` 應為 0);原語有「跨脈絡是否成立」的誠實標註(有的原語只在原設定成立,要講)。

### S8 — 回搬整合(★ 最痛缺口;把發現接進你的系統)
- **掃內部候選**:這些原語能接進哪個現有專案/系統?明確列候選,別讓它懸空(缺口 A 的失敗全是「驗完沒接就忘了」)。
- **最小接線**:import / 移植範本,不重寫;接完跑該系統既有測試確認沒弄壞。
- **寫進 memory**:在你的 agent memory / 筆記系統新增記憶檔(結論 + S7 的觸發條件)+ 索引行。
- **成功判準(嚴格)**:內部系統**真的呼叫到**了這個原語/範本才算整合完成——「存在 memory 裡」不算,「把 INSIGHT.md 加進 repo」也不算(實測:6/11 的失敗多是停在這)。若這篇論文的結論是「不適合我們」,也是合法終點,但要在 memory 明確寫下否決理由。
- **驗收**(擇一,要能舉證):(a) 附得出 diff / commit hash,顯示**某個既有系統**新增了對此原語/範本的呼叫點;或 (b) memory 明載「評估後否決,理由 X」。單純新增 INSIGHT.md 或 memory 檔本身不構成整合。

## 反覆踩的雷 · 全程 checklist(來自 11 個真實專案)

1. **引用先查**:arXiv 標題/作者/id 寫進文件前 WebFetch 核對,查不到標「未查證」。
2. **export≠正確**:功能閘門量測可觀察行為,不只跑通(re-import 量測、對數字)。
3. **複製優先於魔改**:先 1:1 拿乾淨基準,再改;混進既有系統就找不到哪裡壞。
4. **驗證不自驗**:產出者不驗收自己的產出,獨立 reviewer 必開。
5. **計數/排序指標可被篩選條件汙染**:top-N 是篩選結果,不是分佈的證明;要 shuffle / blind 對照才知道現象是真的還是選出來的。
6. **本機限制提早確認**:無 GPU / 無 AVX2 / 無 sudo → 先設計繞法再動手。
7. **參考實作自身可能錯**:驗證鏈是 論文→參考→你的複刻,每個環節獨立確認(「對齊參考≠對齊論文現象」)。
8. **指標好看要兌現下游**:forecast alpha≠portfolio alpha;跑端到端組合/任務測試。
9. **≥3 seed 才算穩健**:單 seed 的方向宣稱常被多 seed 打掉。
10. **散文護欄≠機器邊界**:要移植「安全/隔離」類設計時,認清是「叫 LLM 自律」還是「讓它根本沒手」——移植後者。

## 產出物約定(建議專案結構)

```
<paper-name>/
├── original/           # S2 只讀保留區(clone 原封)
├── repro/              # S3 clean-room 重製
├── gates/              # S4 功能閘門腳本
├── audit/              # S5–S6 驗證報告 + 撤回記錄
├── INSIGHT.md          # S7 三件套(機制圖+原語+觸發條件)
└── PROGRESS.md         # 各階段狀態 + 宣稱清單裁決
```

## 接線你自己的系統(採用者請自行補一段)

本 skill 主體刻意不綁任何特定環境。建議在你的副本末尾加一段「本機接線」附錄,內容:

- **S1 路由**:你環境裡負責深度研究 / repo 分析 / HF 存取的具體工具或 skill。
- **S8 內部候選系統清單**:按領域列出你現有的專案/系統——這是 S8 不懸空的關鍵,清單在手,萃取出的原語才有地方接。
- **memory 落地位置**:你的 agent memory 或筆記系統路徑與索引慣例。
- **先例對照**:走過幾次後,把「S7 原語抽得最乾淨」「S8 回搬最完整」的自家案例記在附錄,當後續範本。
